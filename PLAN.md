# High Level Overview
- This is a journaling app that will be on both Windows and Android, and will sync through a cloud database run on Firestore. Each instance of the app (e.g. 1 on phone, 1 on laptop, 1 on tablet), will store it's own copy of the entire database, along with one firestore database
- Base features include a journal with journal entries, a calendar, to-do list, and global hotkeys (On Windows)
# Features
- Statistic counter: This will be an integer tracker that the user is able to add to every day/week/month/year (This can be also determined by the user). This can also be a boolean tracker (e.g. "Did you shower today" which can only be yes or no), or a dropdown tracker (e.g. "Which gym did you go to" and the answers can be "Gym A", "Gym B", "Gym C", etc). The user can decide between adding an integer tracker, a boolean tracker, or an enum (Dropdown) tracker. The integer tracker should be defaulted to 0 (Unless otherwise specified by the user), the boolean tracker should be defaulted to false (unless otherwise specified), and the enum tracker should be defaulted to empty (For this particular tracker give the user the option of making an option the default). For each statistic, the user is able to decide the color that it is, so it can be shown on the calendar
	- For example, if the user chooses to track how many times they went to the gym and they choose to store this data weekly, at the end of the week, they will be prompted with a page (On either the sunday or monday journal entry, or the most recent journal entry if they do not log on at the start of the week), and they will be prompted to enter the number of times they went to the gym that week
		- Note the starting day of the week should be able to be changed in settings
	- This statistic should have the **option** of being shown on the calendar, both in the widget and in the app. If the statistic is a boolean, then simply show a flag on the calendar, but if it is an integer tracker, using the user's inputted color, display a gradiant, so for higher days use a darker color, and for lesser days use a lighter color
	- The integer tracker should have an option of adding a "cap" so for example you should be able to create a tracker for your energy levels, and you can toggle a cap at 10, so all possible inputs are 0-10, but you should also be able to create a tracker for how many hours studied, and this would have no cap
- Daily/weekly/monthly/yearly rankings: There will be a 1/X scale, where the user is able to choose what value X is, and every day/week/month/year they will be prompted to rate the previous time period. There should be a natural gradient, from color A to color B (which can be edited in settings). 
- Graphs: There should be graphs available for all necessary statistics. If the user adds a new boolean statistic tracker, then they should be able to see graphs that show how many times that flag is true, including how many times per month, per week, how many total times it was flagged this year, and how many net flags over the entire history of the app. Similar to an integer flag, although that will have line charts and such showing the progression over a week, month, year, etc. 
	- Include statistics such as words written, tasks completed, hours studied, hours slept, habits completed, total journal entries, longest daily journal entry streak, and how many total journal entries. 
- Rich text editor: For the journal entry editor, it should support markdown (Along with extended markdown features such as highlighting, etc), and ctrl + z for undoing
- Debouncer: Anywhere where an input is entered, a debouncer will run with an X second timer (Defaulted to 3, should be able to be altered in the code, but not the user settings), and it will update the timer (restart it effectively), when a new input is made. Once the timer runs out, the new updates will be uploaded to the firestore database. This debouncer does not apply to time-sensitive items such as the to-do list
- Calendar: The application will have an option for a calendar page, which shows a calendar. The calendar view can be changed from weekly to monthly to yearly. By clicking on a date, the user is able to add an event. Each event will have a starting and ending time along with a switch for a "full day" event, but the default will be a full day event. Each event can also be a color picked from the color palette and a title, along with a "notes" section that contains text.
	- Offer different heatmaps, such as one for mood, studying, writing, and each statistic. There is a default heatmap, which jsut includes all statistics that the user explicitly chose "Show on calendar" with
- Global keyboard shortcut: The app should offer two shortcuts, which can be customized by the user. One to type into a new journal entry for the current date, in a small pop-up window, not the full application UI. Another shortcut should open up a small popup with the to-do list, and the user should be focused on a new task entry. This feature is only available on windows, and should be implemented using the RegisterHotKey with WinUI 3
	- Note on the pop-up for the to-do list, you should be able to click off tasks completed too
	- Each time a hotkey is invoked, the application should NOT start up, only the small portion necessary to keep the intended behavior. So the app should be focused on starting up quick when a hotkey is pressed (Should never be running in the background constantly, other than listening for the hotkey of course)
- Guided Journaling: User is able to toggle "Guided Journaling" when making a new journal, where there will be a default set of prompts that cycle every X number of days. This is essentially a new journal where whenever the user creates another journal entry the title is autocompleted with a random question. 
	- As an edge case, if the user creates two journal entries on the same day, they will have the same question
- Study tracker: A separate feature, similar to the calendar and to-do list, which tracks how many hours you studied, provides a timer to log studying time, a generated productivity score, and graphs comparing this weeks/months performance to the previous week/month
- Global search: User can search for keywords to display all journal entries with the words in their body or title. Words are separate, so a phrase like "I went to the beach" would pop up if "beach went" was searched since it contains both "beach" and "went". User can also search by tags, and this just shows all entries with the same tag. 
- To-do list: A list that has tasks. User is able to specify metadata, such as date, notes, etc. Each task is able to be completed by checking them off. 
	- User will be able to create different lists, and title them accordingly, along with giving them a custom name
- Life timeline: Another section just like the calendar and journal, but this is for major life achievements/milestones. User is able to add to it freely, and it will display a line in the center with marks along major years, and text to the left and right for each achievement/milestone entered. User is able to select in settings whether it shows the future years (Starting at year 0) or if they enter their age it will show until age 100
- Timed Surveys: User can generate their own surveys that are asked every year, month, and week (Different surveys for each time period), and you can add questions to the survey whenever you want (Although then by default, previous surveys will have "N/A" or 0 or whatever the equivalent is). Once the time period between surveys is done, the user will be prompted to finish another survey.
- Bucket List/Dream List: Separate from the to-do list. This is specifically meant to be used for long term goals. This will be a separate tab from the to-do list
- Quotes: There will be a list of quotes that comes pre-installed (But user is able to add/remove from them), and along with each journal entry every day, a quote will appear in the bottom right corner (Although user is able to turn this off). The quote appearing in the bottom right corner is NOT next, it is part of the background. This quote is saved for future views of this journal entry. Once the user turns off their setting to view the quotes for their journal entries, ALL journal entries will hide their quotes (Although save it in the metadata in case they turn the quotes back on). Note that if the user has quotes turned off, each journal entry still has a quote tagged alongside that entry (In case the user turns on quotes again later). Quotes are pulled from the bank at random (Although every quote must appear once before the bank is reset)
	- User should also have the option of overriding the quote with their own quote of the day
# Frontend UI
- For every journal entry, there will be a title. Then there will be some input boxes for relevant data, such as statistics (Only the necessary statistics are shown, so for example daily statistics will always be shown, but weekly statistics are just shown at the start of the week, or the next time the user is online after a week reset)
- At the very top of the journal entry text box, if the user enters any hashtags (anything starting with a "#"), that tag will be colored to a unique color, and stored, so that any future invocation of the same tag will be the same color
- The journal entry text box should follow markdown (So allow for # for titles or ** for bolding or == for highlighting, etc), but should NOT support font changing or text sizing. 
	- The title should not have any formatting options
- At the very left of the screen should be an icon bar with all the icons (Ex. Finances, journal entries, calendar, settings, etc), and to the right of that (but still on the left of the screen) should be all the other journal entries (this only applies to the journal entry screen)
- At the top right of the screen (For the whole app) should be the time, in the style of a clock with physical pages that flip past (animated), and a small icon for the current weather.
- Android app's UI should mimic that of the desktop
## Theme
- Minimalistic, defaults to dark with colored accents (Which can be chosen by user)
- Icons are also minimalistic, with some gradients
- App itself will have a slight paper texture, and include gradients between the chosen theme (Light or dark), and the accent specified by the user.
# Low Level Design
- Both Windows and Android apps will be written using Flutter
- All data will be stored on EACH device with the app instance (e.g. If i have this app on my phone, tablet, and laptop, I will have a copy of all the data on EACH device so I will have 3 local copies). Additionally, any changes will be uploaded to firestore. Every time a device opens up an instance of the app, it will update it's local database to incorporate the firestore's changes locally
	- When syncing between instances, there will be a local embedded database on each device. Android will use Room and Windows will use a local SQLite database. The app will operate on a client-heavy, serverless model
		- Journal portion: When you type a journal entry locally, it will save directly to the local device's hard drive. Then, using a debouncer, once you stop typing for 3 seconds (This resets every keystroke, so if you type a letter, wait a second, then type again, the debouncer will reset back to 3 seconds and nothing will be pushed yet). Once the debouncer hits zero, it will copy the finished text to the cloud database. It will copy the whole current document. Later when another app instance opens up, the device will ping the cloud database to see if any updates happened, and if so, it will update it's local storage before showing the screen 
			- Note that this debouncing timer should be a variable (Thus it should be able to be altered)
			- Note that every time a user updates a journal entry (or any other entry type), that will be stored as **one document**, and each document will be for each day (So if I make a journal entry on day 1, then one on day 2, I will have two documents. On the third day, if I both update day 1's entry, and add a day 3 entry, that device will upload onto the cloud database documents 1 (from day 1, and the entire document), and document 3 (from day 3))
				- Each document does not have to be limited to just journal entries. For example, one journal entry document will contain information such as metadata (Time and date), any tags used, etc. There will also be documents for miscellaneous data (e.g. A financial statements document every day which stores all relevant financial data, and another document that tracks the various statistics a user can choose to track, like how much sleep they got or how much studying was done)
		- Live Portion: For items such as the to-do list, which must be updated instantaneously use a different protocol. When you open the to-do list on multiple devices, each device will open their own continuous, live connection the cloud database. They will sit and listen for any updates. Then once an update is registered, such as checking off a task or adding a new task, the device that made the update will tell the cloud, and the cloud will then instantly push to the open connections
- The app calendar should pull any upcoming events from Google Calendar as well, but this will be a READ-ONLY design, so any events added to the app calendar should NOT be reflected on the user's Google calendar
	- To do this, the last app that the user uses will sync into the user's Google calendar and read any new events, then it will take those events and overwrite them or update them in the central firestore database, then when other devices open up later, it will simply pull those records out of the firestore. This way there does not need to be a dedicated backend that pulls from Google calendar
- For any feature that the user must update every X number of days (e.g. weekly or monthly or yearly), the app should show those option boxes when the user needs to update them (Like showing yearly options on January first), but also create an INTERNAL feature that allows the app to instead notify the user on the proper day, or whenever they open the app (so now the user will be notified on January 1, at whatever time they specify alerts should be at, and the next time they open the app they will be prompted with a toast notification that tells them they have to enter in whatever they need to enter), but default this flag to false for now. If the user does NOT log in at the necessary date, then still prompt the user with the option the next time they log in (e.g. if the user misses new years, and their next login is on jan 5, you should still prompt them for the entries they needed to have made from jan 1 to jan 5) 
- Add Firebase Authentication (Email/Password or Google OAuth) 
- For conflict resolution between two devices that both edited the same file, use CRDTs (Conflict-free Replicated Data Types), specifically sequence CRDT that is operations-based.
- Utilize soft deletes, so once an object is marked as deleted, store a flag and the timestamp of when it was marked as deleted, and only delete once that timestamp passes 30 days.
- Upon app startup, implement lazy loading so only the necessary and relevant data is loaded up at startup, but once the user tries to access older data, that data is pulled too (From the local database)
- To allow for the windows hotkey using Flutter, use a package like hotkey_manager and design the app to run headless (without a window) until prompted by either clicking the icon or using the shortcut.
- Give every journal entry a unique UUID, then make the date a metadata field inside the document, so when a user changes an entry from day 2 to day 1, you overwrite the date field, but the full journal entry can still be uploaded to the firestore database without worry of duplication.
# Miscellaneous
- If a user has a journal entry made on day 1 and day 2, but then edits the day 2 entry to be instead on day 1, then day 1 would have TWO journal entries. On the cloud, you should upload only the newly added document, not the old document
- There should be an option to add multiple "journals" which all store their own unique entries. So for example there can be a school journal and a personal journal, and each journal will store their OWN entries every day, and they will not mix. On the journal page (to see all journal entries), at the top right there should be a dropdown menu (Only appears if there is MORE than one journal), and it should allow you to switch between specific journals, or just a "all journals" overview
- Journal entries should have a timestamp but this timestamp is editable by the user, and otherwise defaults to the **creation date**, not the last edited date. Entries also contain weather auto-tagging, symbolized by a small icon. User is able to mark their mood out of 10 for the day. A single entry can be categorized through tags. User can track additional statistics in an entry by adding that to their statistic tracker, such as hours slept or energy levels. These statistics should be able to be viewed as a graph.
- Text boxes must support undoing, and any deletion of journal entries (Or other files), should be stored in the recycling bin for 30 days. 
- Allow for easy support of future plugins
- To prevent race conditions reading the user's google calendar, use a lock mechanism in the Firestore database, where a device must claim syncing rights, then sync with the calendar
